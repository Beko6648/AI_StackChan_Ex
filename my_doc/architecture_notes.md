# アーキテクチャ検討メモ

---

## スタックちゃんと Claude Code の接続構想（検討・却下）

### 構想の概要

スタックちゃんに話しかけるだけで、ファームウェアの修正・ビルド・書き込み・動作確認まで完結できる「自己進化型デバイス」を実現できないか検討した。

---

### 検討した方式と結果

**方法A: Claude Code → WebAPI → スタックちゃん（Claude Code が脳）**

Claude Code がスタックちゃんの WebAPI（/speech / /face / /servo_offset など）を呼び出してハードウェアを制御する。スタックちゃんを Claude Code のアクチュエータとして使う構成。

技術的には実現可能。ただし：
- スタックちゃんが「自律したAIマスコット」から「Claude Code の身体」に変わる
- 感情システム・MoodManager・キャラクターシステム・自発発話・就寝サイクルなど、これまで実装した全ての「知性」部分が不要になる
- PC が常時起動している必要がある
- ユーザーの音声を PC に届ける手段が別途必要
- ファームウェアをほとんど更新する必要がなくなり、開発の意義が変わってしまう

→ **現在の「自律型AIマスコット」という設計思想と相反するため却下**

---

**方法B: スタックちゃん → Anthropic API（Claude を LLM バックエンドに追加）**

ChatGPT の代わりに Claude API を呼び出す LLM バックエンドを実装する。キャラクターシステムとの両立は問題なし。

ただし：
- Pro サブスクリプションとは別に Anthropic API の料金が発生する
- gpt-4o-mini と比較してコスト面でのメリットがない

→ **コスト面の理由で却下**

---

**方法C: Function Calling + Claude Code Remote Control（本命案として検討）**

gpt-4o-mini の Function Calling で Claude Code の Remote Control を呼び出し、ファームウェア変更を依頼する。Pro サブスクリプション内で開発処理を完結できる可能性があると考えた。

調査の結果：
- Claude Code の Remote Control は HTTPS ポーリング（アウトバウンドのみ）
- ESP32 などの外部クライアントからの HTTP リクエストには対応していない
- 処理結果を構造化したレスポンスとして受け取ることもできない
- 対応クライアントは VS Code・claude.ai/code・モバイルアプリのみ
- 認証は claude.ai サブスクリプション必須（API キー不可）

→ **技術的に実装不可能なため却下**

---

### 根本的な設計判断

「Claude Code が脳・スタックちゃんが身体」の構成を採用すれば技術的な問題は解決するが、**スタックちゃんを「自律したAIマスコット」として育てる**という当初の目標と根本的に相反する。

現在の実装路線（自律型AIマスコット）を継続することが最適という判断に至った。

---

### 将来の再検討条件

以下のような状況変化があれば再検討の余地がある：

- Claude Code が外部クライアント向けの HTTP API を公開した場合
- ESP32 から呼び出せる軽量な AI エージェント API が登場した場合
- PC 常時起動のコストが許容できる運用体制になった場合
