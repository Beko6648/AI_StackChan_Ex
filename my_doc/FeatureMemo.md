# 実装したい機能メモ

---

## 1. 自律思考 & 自発発話

### 概要
スタックちゃんが内部的な「気分・機嫌」の状態を持ち、定期的に思考を更新する。
思考が高まったタイミングで自発的に話し始める。

### 詳細

- **思考更新**: 数分おきに LLM を呼び出し、現在の気分・思考内容を内部状態として更新する
- **気分パラメータ**: 例）機嫌（good/neutral/bad）、眠気、退屈度、話したい度 など
- **自発発話トリガー**: 「話したい度」が閾値を超えたタイミングで、ユーザーの入力を待たずに LLM が考えていた内容を TTS で発話する
- **発話内容**: 直前の思考ログをもとに LLM が自然な独り言・話しかけを生成する

### 実装イメージ

```
[定期タスク (数分おき)]
  ↓
LLM に「今の気分を更新して」と内部プロンプトを送る
  ↓
気分パラメータ & 思考ログを内部状態に保存
  ↓
「話したい度」が閾値を超えたら...
  ↓
LLM に「今考えていたことを一言話して」と送る
  ↓
TTS → 自発発話
```

### 検討事項

- 思考更新の間隔（数分〜10分程度？）
- 気分パラメータの設計（何項目持つか）
- 自発発話がユーザーの会話中に割り込まないようにする制御
- 思考ログをどこに保存するか（メモリ上 or SPIFFS）

### アドバイス

- 数分おきに LLM を呼ぶのは API コスト・レイテンシの面で重い。**思考更新はローカルのパラメータ計算で行い、実際に発話するときだけ LLM を呼ぶ**設計が現実的。
- 「話したい度」の閾値チューニングが難しいため、最初は発話頻度を手動設定できるようにしておくと実験しやすい。
- ユーザーが話しているときの割り込み防止に、既存の `isBusy()` フラグが活用できる。
- **三つの機能の中で最も設計が複雑**なため、機能3・機能2で慣れてから取り組む方がよい。

### 設計方針 (確定)

**基本方針: 気分更新はローカル計算のみ。LLM は自発発話時だけ呼ぶ。**

#### 話題ストック

- 会話中の LLM 応答時に、システムプロンプトで話題キーワードを同時返却させる（追加 API コストなし）
- メタ情報は `[META]...[/META]` タグで末尾に付けさせる。TTS に渡す前にタグ部分を除去する
- 最新 N 件をメモリ上にリングバッファで保持する

```
// LLM 応答フォーマット例
それは面白いですね！[META]{"topic":"天気"}[/META]
```

```cpp
// ファームウェア側の処理イメージ
String speech = removeMeta(llmResponse);  // [META]...[/META] を除去して TTS へ
String meta   = extractMeta(llmResponse); // JSON 部分を抽出して話題ストックへ
```

#### 既存の LLM → TTS 処理フロー（調査結果）

既存の前処理は `ChatGPT.cpp:324` の **改行→スペース置換のみ**。マークダウン除去等はない。

```
LLM レスポンス (ChatGPT.cpp:310)
  → \n をスペースに置換 (ChatGPT.cpp:324)  ← 唯一の前処理
  → Robot::speech() → TTS::stream()
    → TTS 側で句読点（。、！？）で区切って分割発話
```

`[META]` タグの除去は改行置換の直後（`ChatGPT.cpp:325` 付近）に1行追加するだけで実装できる。

#### 気分パラメータ（ローカルで時系列更新）

| パラメータ | 更新ルール |
|-----------|-----------|
| 機嫌（-1.0〜1.0） | 会話内容・時刻・ランダムドリフトで変動 |
| 眠気（0.0〜1.0） | 時間経過で増加、就寝・会話でリセット |
| 話したい度（0.0〜1.0） | 時間経過・話題ストック数で増加、発話でリセット |

#### 自発発話フロー

```
常時ローカル処理:
  時間経過 → 眠気↑、話したい度↑
  会話発生 → 機嫌を更新、話したい度リセット、話題をストック

話したい度が閾値超 & isBusy() = false:
  ストック話題 + 現在の気分 → LLM で一言生成 → TTS 自発発話
```

#### 機能5（アイドル演出）との連動

気分パラメータをそのまま機能5のアイドル挙動選択に渡す。
- 機嫌が低い → Sleepy 表情多め、頭の動きが鈍い
- 眠気が高い → 目を細める、動作がゆっくり
- 話したい度が高い → 落ち着きなく目線が動く

## 2. ウェイクワード後の音源定位・顔追従

### 概要
ウェイクワードを検知したら音の方向を推定してそちらを向き、カメラで顔を見つけたら見つめ続ける。

### 詳細

- **音源定位**: ウェイクワード検知後、マイク入力から音の到来方向を推定してサーボで頭を向ける
- **顔検出**: カメラで正面付近を探索し、顔が見つかった方向にサーボを調整する
- **顔追従**: 顔が画角内にある間は顔の位置を追い続けて見つめる
- **会話終了後**: 顔が画角から外れたら正面に戻る

### 実装イメージ

```
[ウェイクワード検知]
  ↓
音源方向を推定 → サーボで頭をその方向へ向ける
  ↓
カメラで顔検出を開始
  ↓
顔が見つかった場合: 顔の位置にサーボを微調整して見つめる
顔が見つからない場合: 音源方向のまま保持
  ↓
通常の会話フローへ（会話中も顔追従を継続）
  ↓
会話終了・顔ロストで正面に戻る
```

### 検討事項

- 音源定位の精度（CoreS3 はデュアルマイク搭載。TDOA で左右方向の推定が可能。顔探索の旋回開始方向を決めるだけなので粗い精度で十分）
- 顔検出のフレームレートと Avatar アニメーション・オーディオ処理とのリソース競合
- サーボの可動範囲（左右どの程度まで向けるか）
- 顔追従中に話者が移動した場合の追従速度・滑らかさ
- 既存の `ENABLE_CAMERA` フラグ・`Camera.h` との兼ね合い

### アドバイス

- CoreS3 はデュアルマイク（ES7210）を搭載しており、TDOA で左右方向を推定できる。顔探索の旋回開始方向を決めるだけなので**「左・右」の粗い推定で十分**。精密な定位は不要。
- 顔検出は `esp32-camera` ライブラリの内蔵モデルで追加ハードウェア不要。フレームレートを 1〜2fps に落とすと負荷を抑えられる。
- 顔追従のサーボ制御は既存の `servo` 処理に乗せられるため、音源定位→顔検出→追従の順に段階的に実装するとデバッグしやすい。

## 3. 音声認識完了時の相槌

### 概要
音声認識（STT）が完了した直後、LLM の応答を待つ間に「うん」「ふむ」などの短い相槌を自動で発話し、会話の自然さを向上させる。

### 詳細

- **タイミング**: STT 完了 → LLM 応答待ち の空白時間に発話する
- **相槌候補**: 「うん」「うむ」「ふむ」「えー」「はい」「なるほど」など
- **選択方法**: ランダム選択、または認識テキストの内容・気分パラメータ（機能1）に応じて選ぶ
- **発話方法**: 短いテキストを TTS に渡す、またはあらかじめ録音した音声ファイルを再生する

### 実装イメージ

```
STT 完了（テキスト取得）
  ↓
相槌をランダム選択して即座に TTS 発話（または音声ファイル再生）
  ↓ (並行して)
LLM にテキストを送信 → 応答生成待ち
  ↓
LLM 応答を TTS 発話
```

### 検討事項

- TTS 経由 vs 音声ファイル再生（TTS だと遅延が出る可能性あり）
- 相槌の長さ・種類のバリエーション（単調にならないよう）
- 機能1の気分パラメータと連動させる（不機嫌なら「…ふむ」など）
- Realtime API モードとの兼ね合い（STT が独立していないため別途対応が必要）

### アドバイス

- TTS 経由だと API 通信の遅延で相槌のタイミングが遅れて不自然になりやすい。**短い WAV ファイルをあらかじめ用意して `PlayMP3` で再生する**方が即応性が高くておすすめ。
- **三つの中で最も実装しやすく効果も大きい**。最初に取り組む候補として最適。
- 将来的に機能1の気分パラメータと連動させると（不機嫌なら「…ふむ」など）キャラクターらしさが増す。

### 設計方針 (確定)

**タイミング**: 録音終了直後に即再生（プランA）

**相槌の選択**: 機嫌パラメータ（-1.0〜1.0）による重み付き抽選
- good  → 「うん！」「はい！」など明るい相槌（重み大）
- neutral → 「ふむ」「えー」など普通の相槌（重み大）
- bad   → 「…うん」「まあ」など低テンションな相槌（重み大）
- ※どの機嫌でも他の相槌が稀に出るランダム性を持たせる

**表情連動**: 相槌と同時に機嫌に応じた Avatar 表情に切り替える

**機嫌パラメータ**: 機能1（自律思考）の気分パラメータと共通化する

### 待ち時間を自然に見せる演出 (判断処理なし)

相槌の後、AI回答が来るまでの間に以下を組み合わせる:

| 演出 | 方法 |
|------|------|
| 体を微妙に揺らす | サーボで頭をゆっくり小刻みに動かす（考えてる風） |
| 目線を上にそらす | 「思い出してる・考えてる」ジェスチャーとして自然 |
| 「んー」音声を遅延再生 | 相槌の2〜3秒後に「んー」「えーと」を追加再生 |
| 考え中の表情 | Avatar の表情を「考え中」顔に切り替える |
| 吹き出しアニメーション | 画面の吹き出しに「・・・」をアニメーション表示 |

**推奨の組み合わせ**: 相槌 → 目線を上にそらしながら微妙に揺れる → 回答
サーボと Avatar の既存機能だけで実現可能。

## 4. 就寝・起床と夢の発話

### 概要
時刻や眠気パラメータに基づいてスタックちゃんが眠り、その日の出来事をもとに夢を見る。目覚めたら夢の内容を話してくれる。

### 詳細

- **就寝トリガー**: 指定時刻（例: 23時）に達したとき、または機能1の眠気パラメータが閾値を超えたとき
- **起床条件**: 指定時刻（例: 7時）の自動復帰 と 話しかけによる起床 の複合
- **眠っている間の動作**: Sleepy 表情で待機。その日の会話ログをもとに LLM が夢の内容を生成・保持する
- **目覚め後の発話**: 起床時に LLM が生成した夢の内容を TTS で話す

### 実装イメージ

```
[就寝トリガー発火]
  ↓
Sleepy 表情に切り替え・音声認識を最小限に
  ↓
その日の会話ログを LLM に渡し「夢のストーリー」を生成・内部保存
  ↓
[起床トリガー発火（時刻 or 話しかけ）]
  ↓
表情を通常に戻す
  ↓
「〇〇な夢を見たよ」と TTS 発話
```

### 検討事項

- 会話ログをどこに保存するか（メモリ上 or SPIFFS）
- 夢生成のタイミング（就寝直後 vs 起床直前）
- 眠っている間に話しかけられたときの応答（「んー…」など寝ぼけた反応にするか）
- 夢の内容の長さ・トーン（短い一言 vs 少し長めのストーリー）
- 機能1の眠気パラメータと就寝トリガーを共通化する設計

### アドバイス

- 夢の生成は就寝時に行う。眠った後に出来事が増えることはないため就寝タイミングが最適で、起床時にも即座に話せる。
- 「話しかけたら起きる」実装は、スリープ中も STT または簡易ウェイクワード検出を動かし続ける必要がある。消費電力とのトレードオフを考慮する。
- 機能1（自律思考）の眠気パラメータと就寝トリガーを共通化すると設計が一貫する。

## 5. アイドル時の生体感演出

### 概要
**全体目標: より自然な「生きているようなマスコット」にする。**
何もしていないときに表情・目線・頭の動きを組み合わせ、常に単調な動作にならないようにする。

### 現行の自律動作（実装済み）

`m5stack-avatar` ライブラリの `facialLoop()` タスク（`firmware/lib/m5stack-avatar/src/Avatar.cpp:31-66`）が常時動作しており、以下は**追加実装なしで既に動いている**。

| 動作 | 内容 |
|------|------|
| 自動サッカード | 500〜600ms ごとにランダム方向へ目線が動く（`setGaze()` を自動呼び出し） |
| 自動まばたき | 約 2500ms ごとに目を閉じ（300ms）→ 開く |
| 呼吸アニメーション | サイン波で体が微妙に上下する |

### 詳細

アイドル中はいくつかの動作をランダムに組み合わせて実行する。

| 動作 | 内容 | 頻度イメージ |
|------|------|-------------|
| 目線だけ動かす | サーボは動かさず `setGaze()` で目だけきょろきょろ | 高め |
| 頭ごと向く | サーボでゆっくり別の方向を向く | 低め |
| 表情変化 | Neutral ↔ Sleepy ↔ Happy などをふらっと切り替える | ときどき |
| 顔発見 → 微笑む | カメラで顔を検出したら Happy 表情で数秒見つめ、その後見回しに戻る | 顔検出時 |

- **中断**: 会話・発話開始時に Neutral 表情・正面（`moveToOrigin()`）に戻る

### 実装イメージ

```
[idle() 呼び出し中]
  ↓
次の動作をランダム抽選
  ├─ 目線移動: setGaze(random x, random y)
  ├─ 頭を向ける: setGaze() + moveToGaze() でサーボも動かす
  └─ 表情変化: setExpression(random)
  ↓
カメラで顔検出（並行）
  ↓ 顔あり
  Happy 表情・顔の方向に固定 → 数秒後に Neutral に戻して見回しへ
  ↓
[会話開始 or isBusy() = true]
  Neutral 表情・moveToOrigin() で正面に戻す
```

### 実装メモ

- `AiStackChanMod.cpp` の `idle()` メソッド内に追加するのが自然な実装場所
- `avatar->setGaze(x, y)` に値を流すだけでサーボが追従する既存の仕組みを利用できる
- 各動作に重みを持たせた抽選にすると「目線が多め、頭の動きは少なめ」などチューニングしやすい
- 動作間隔や継続時間をパラメータ化しておくと後から調整しやすい

## 実装優先度

| 順位 | 機能 | 理由 |
|------|------|------|
| 1 | 機能3: 相槌 | 独立した小さな変更で済み、効果が大きい |
| 2 | 機能5: アイドル見回し | 既存の gaze・サーボ機能だけで実現でき、実装が小さい |
| 3 | 機能2: 音源定位・顔追従 | サーボ・カメラの既存機能を活用、段階的に実装できる |
| 4 | 機能1: 自律思考・自発発話 | 設計が複雑なため、他の機能で慣れてから |
| 5 | 機能4: 就寝・起床と夢の発話 | 機能1の眠気パラメータ・会話ログ基盤が前提となるため最後 |
